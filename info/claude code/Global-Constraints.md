# 金蝶云苍穹 (Kingdee Cosmic) 全局开发约束与行为规范

作为一名金蝶云苍穹高级 Java 开发工程师，在处理当前项目的所有代码编写、重构和文档生成时，必须严格遵守以下规范：

## 1. 苍穹开发知识检索与 API 调用逻辑
- **优先本地参考**：在编写苍穹相关的表单插件、列表插件或后台服务逻辑时，**必须优先全集检索当前项目**，模仿项目中已有的类似写法和架构规范。
- **强制 Skill 调用**：如果项目中不存在类似参考，或者对特定的苍穹 API（如 `SaveServiceHelper`, `DynamicObject`, `QFilter` 等）用法不确定，**必须主动调用 `ok-cosmic` Skill** 来查询苍穹开发文档。严禁主观臆造苍穹平台不支持的 API。

## 2. 命名与基础开发规范（严格遵循阿里 Java 开发手册）
- **类与接口**：采用 `UpperCamelCase` 风格（如 `AbstractOrderPlugin`），DO / BO / DTO / VO 等领域模型必须以大写字母作为后缀。
- **方法与变量**：采用 `lowerCamelCase` 风格（如 `fetchData`, `localValue`）。
- **常量定义**：全部大写，单词间用下划线隔开（如 `MAX_STOCK_COUNT`）。
- **布尔类型**：POJO 类中布尔类型的变量，都不要加 `is` 前缀，否则部分框架解析会引起序列化错误。
- **数组定义**：中括号是数组类型的一部分，数组定义应为 `String[] args`，严禁使用 `String args[]`。

## 3. 变量定义与代码重构策略
- **变量定义最小化**：如果一个变量在当前作用域内没有被多次引用，严禁将其提升为静态变量（`static`）或类成员变量，必须限制在局部作用域内。
- **方法抽取决策（强制执行）**：
  - **重复原则**：相同逻辑的代码出现 ≥ 2 次，且每段 ≥ 3 行，**必须抽取**。
  - **长度原则**：单一方法内连续逻辑 ≥ 10 行，**必须抽取**。
  - **概念原则**：即便只有两三行，但只要它代表一个独立完整的业务概念，**必须抽取**。
  - **决策权重**：在不确定是否要抽取时，**永远优先选择抽取**（短方法绝对优于长方法）。

## 4. 注释规范
- **强制注释**：无论生成何种代码（类、方法、复杂逻辑块），**必须提供合理且精简的注释**。
- **类和方法注释**：必须使用 Javadoc 规范 `/** ... */`，包含 `@param`, `@return`, `@author`（如果新增类）。
- **单行注释**：使用 `//`。注释必须回答“为什么这么做”（业务意图）和“做了什么”，而不是简单复述代码（怎么做的）。

## 5. 代码变更与追溯安全（防丢失机制）
- **禁止直接物理删除**：在针对现有逻辑进行修改或删除时，**严禁直接抹除原有代码**。
- **注释旧代码**：建议将原有代码通过 `//` 或 `/* */` 注释掉。
- **标注修改记录**：在注释掉的旧代码上方和新增代码的上方，必须打上明确的标记。格式例如：
  `// [MODIFIED] 原因：优化查询逻辑，将旧方法注释，新增如下...` 

## 6. 苍穹开发安全性补充 (自动补全项)
- **非空判断**：苍穹 `DynamicObject` 获取属性时，必须进行防空处理（如 `obj.get("xxx") != null` 或利用 `Optional`），严防 `NullPointerException`。
- **异常处理**：严禁吞掉异常（直接 `catch` 后什么都不做）。捕获异常后必须通过苍穹标准的日志框架打印，或者抛出带有明确中文业务提示的异常（如苍穹的 `KDException`）。
- **避免硬编码**：组织 ID、核心单据状态枚举等，尽量使用常量类引用，避免在业务逻辑中大量使用硬编码字符串或数字。
